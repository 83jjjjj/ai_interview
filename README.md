做这个项目是出于两个方面，一是锻炼sdd即规格驱动开发思想的ai coding能力，二是近期多次面试表现欠佳，心血来潮做个模拟面试平台，既可以自己用，也可以便利他人。
下面记录项目开发流程，以下绝大部分流程要么ai辅助，要么ai全部：
1. 我先模糊地表达出需求，进而拆分成较为细致功能模块，在ai不断review的过程中产出spec/requirements.md
2. 让ai根据需求规格文档，给出可选框架与多维度对比，我做决策；
3. 架构设计由ai提出，我review
4. ai输出plan，我review

数据库选型
你需要存的数据：

  - 用户账号（用户名、密码哈希、邮箱）
  - 简历文件（PDF/图片）
  - 面试记录（对话原文、时间戳、评价结果）
  - 面试配置（岗位、风格、难度）

  这些数据之间有明确的关联关系（一个用户有多次面试，一次面试有多条对话记录），是结构化数据。

  ├──────────────┼──────────────────────────────┼───────────────┼──────────────────────┤

  ├──────────────┼──────────────────────────────┼───────────────┼──────────────────────┤

  ├──────────────┼──────────────────────────────┼───────────────┼──────────────────────┤
  │ 并发能力     │ 强                           │ 中等          │ 弱（单文件锁）       │
  ├──────────────┼──────────────────────────────┼───────────────┼──────────────────────┤
  │ 部署复杂度   │ 需要独立服务                 │ 需要独立服务  │ 零配置，文件即数据库 │
  ├──────────────┼──────────────────────────────┼───────────────┼──────────────────────┤
  │ 适合         │ 复杂查询、大数据量           │ 常规 Web 应用 │ 开发阶段、小项目     │
  └──────────────┴──────────────────────────────┴───────────────┴──────────────────────┘

  你的项目 MVP 阶段 50-100 人并发，数据量不大。但你的面试评价模块涉及 JSON 格式的结构化输出（评分维度、改进建议列表），PostgreSQL 对 JSON
  的支持最好。

  建议：开发阶段用 SQLite，部署时切 PostgreSQL。

  理由：
  - SQLAlchemy（Python 的数据库 ORM）支持同时兼容 SQLite 和 PostgreSQL，代码不用改，只改连接配置
  - 开发阶段不用装数据库服务，写代码直接跑
  - 上线时换 PostgreSQL 一行配置的事

  你正在学 LangChain，但这个项目的 AI 交互模式其实很清晰：
  用户消息 → 拼接 prompt（简历+岗位+上下文+用户回答）→ 调 LLM → 返回问题/评价
  这就是单轮/多轮对话，不需要 LangChain 的 Agent、Chain、Memory 等复杂抽象。直接调 LLM API 更简单、更可控、更容易调试。
理由：
  - 你的 prompt 模式固定（面试提问、评分），不需要 LangChain 的动态链
  - 流式输出直接用 API 原生支持，不需要额外封装
  - 出了问题能直接看到发给大模型的是什么，调试成本低
  - 面试时能讲清楚"我选择最简单的方案，避免过度工程化"

实时通信方案：

  ┌────────────┬────────────────────────┬───────────────────────────┬──────────────────────────┐
  │    维度    │       WebSocket        │ SSE（Server-Sent Events） │        HTTP 轮询         │
  ├────────────┼────────────────────────┼───────────────────────────┼──────────────────────────┤
  │ 通信方向   │ 双向                   │ 服务器→客户端（单向）     │ 客户端→服务器（单向）    │
  ├────────────┼────────────────────────┼───────────────────────────┼──────────────────────────┤
  │ 实现复杂度 │ 高（需要管理连接状态） │ 低（就是 HTTP 长连接）    │ 最低                     │
  ├────────────┼────────────────────────┼───────────────────────────┼──────────────────────────┤
  │ 流式输出   │ 天然支持               │ 天然支持                  │ 不支持（只能等全部返回） │
  ├────────────┼────────────────────────┼───────────────────────────┼──────────────────────────┤
  │ 断线重连   │ 需要自己实现           │ 浏览器原生支持自动重连    │ 天然无状态，不存在断线   │
  ├────────────┼────────────────────────┼───────────────────────────┼──────────────────────────┤
  │ 适合场景   │ 聊天室、协同编辑       │ 通知推送、AI 流式回复     │ 简单数据查询             │
  └────────────┴────────────────────────┴───────────────────────────┴──────────────────────────┘
建议：SSE。
  理由：
  - 你的场景是"用户发一条 → AI 流式回复"，不是真正的双向实时聊天，不需要 WebSocket
  - FastAPI 原生支持 SSE，几行代码就能实现
  - 浏览器端用 EventSource API 接收，前端实现简单
  - 断线重连浏览器自动处理
